02

Skill 的工作原理

20 分钟

Skill 的本质

就是一个文件夹

# 一个最简单的 Skill
my-skill/
├── SKILL.md        # 必需：描述 + 指令
├── template.md     # 可选：模板
├── examples/       # 可选：示例
└── scripts/        # 可选：脚本

没有黑魔法。就是 Markdown + 文件夹。

--- SKILL.md ---
---
name: code-review
description: "评审代码：关注安全、
性能、可读性。在用户说'review'
或提交 PR 前自动建议。"
---

# Steps
1. 读取改动的文件
2. 检查 SQL 注入、XSS
3. 检查异常处理
4. 输出 markdown 报告
触发机制

Skill 是怎么"被找到"的？

🔍

描述驱动

Claude 读所有 Skill 的 description，判断哪个最匹配当前任务

⚡

显式调用

你直接打 /skill-name，无需 Claude 判断

💡

主动建议

Claude 看场景觉得合适，会主动提议："要不要用 /qa？"

"好的 description = Skill 在该出现的时候出现，不该出现的时候沉默。" —— 这是写 Skill 最难也最关键的一件事
本质区别

Skill ≠ 长一点的 Prompt

维度Prompt（提问）Skill（技能）
生命周期一次性可复用、长期存在
版本管理没有Git 管理，可回滚
分享复制粘贴开源、安装、订阅
触发你说才执行AI 自己判断 + 显式调用
组织属性个人资产团队资产

动手实战：写一个自己的 Skill

30 分钟

第一步：选题

找一个你"每周都要做 3 次"的事

📊

周报生成器

把一周的 Git 提交整理成可读的周报

✏️

中文文案润色

按公司语调润色英→中翻译

🎓

课件出题器

给一段讲稿自动生成 5 道课堂练习

📧

邮件回复模板

客户咨询自动起草专业回复

📝

会议纪要

把语音转录整理成 action items

🐛

Bug 复现脚本

把 bug 描述变成最小复现 demo

第二步：SKILL.md 三要素

这三件事决定 Skill 好不好用

🏷️

name

简短、可记、kebab-case

weekly-report ✓

my_super_helper ✗

📣

description

最关键的一行

明确说"什么时候用 + 用来做什么"

📋

正文

步骤、约束、输出格式

越具体越稳定

现场跟写

一起写一个 "周报生成器"

---
name: weekly-report
description: "读取最近 7 天的 git 提交，生成中文周报。
当用户说'写周报''生成周报'或周五下午调用。"
---

# 步骤
1. 运行 `git log --since="7 days ago" --oneline`
2. 按"功能 / 修复 / 重构"三类归并
3. 用中文输出，每条不超过 20 字
4. 末尾加一句"下周计划"，向用户提问获取

# 约束
- 不要输出 commit hash
- 不要写英文术语，找不到对应中文就保留原词
- 输出格式：Markdown，三级标题
常见坑

这三个错误九成新手都会犯

📡

description 太宽泛

"帮助用户" → 啥场景都被它抢戏

✓ 改成"读 git log，生成周报"

🔒

description 太狭窄

"周五下午 5 点写周报" → 几乎不会触发

✓ 加触发词"周报/总结"

🌫️

指令含糊

"风格要好" → 每次输出都不一样

✓ 给出格式 / 字数 / 示例

06

进阶概念

10 分钟

作用域

Skill 放在哪儿，谁能用？

位置路径谁能用典型场景
项目级 .claude/skills/ 仅当前项目 团队规范、项目特定流程
用户级 ~/.claude/skills/ 所有项目 个人工具集
插件级 通过插件管理器安装 看插件配置 第三方 Skill 包（如 gstack）
协同

Skill 真正的威力是组合

单 Skill

解决一个具体问题

→ 像一把锤子

Skill × Skill

一个调用另一个，串成流水线

→ 像一条生产线

# 例：/ship 这一个命令，内部串了：
git status  → 看改了什么
/review     → 自动评审
run tests   → 跑测试
bump VERSION → 更新版本号
git commit + push + PR → 一气呵成

这就是下一章 gstack 的核心思想。

拆开看看

一个 gstack Skill 长什么样？

现场打开 ~/.claude/skills/gstack/，挑一个看：

# /office-hours 的 SKILL.md（节选）
---
name: office-hours
description: "YC Office Hours: 用六个强制问题
拷问需求真实性、现状、最窄切入点……
当用户描述新想法、问'值不值得做'时触发。"
---

# 六个强制问题
1. 谁会因为这个 desperate 地付钱？
2. 他们现在怎么解决这个问题？
3. 最窄的 wedge 是什么？
4. 你观察到了什么别人没看到？
5. 这个产品在 5 年后还成立吗？
6. 如果只能做一件事，是哪件？

看，没有黑魔法。就是把 YC 的方法论写成 Markdown。

四个启示

gstack 教会我们什么？

①

流程 > 工具

23 个无序 Skill ≠ 一个有序流程。Skill 真正的价值是组合成工作流。

③

可分享可复用

一个人的工作流变成 MIT 资产，全行业受益。

④

个体的杠杆

Tan 自称生产力 ×810——不是打字快，是会编排 AI 代理。
